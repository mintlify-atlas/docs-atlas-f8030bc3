Friends are the accountability layer in Klariti. You don't build a network — you add the small number of people you actually want to focus with. Shared challenges only create meaningful pressure when the group is real. The friendship system is designed to be low-friction for genuine connections while handling edge cases like re-adds and request withdrawals cleanly and without leaking social information.

## Two-table design

Klariti's friendship model uses two separate tables: `Friendship` (canonical relationship state) and `FriendRequest` (per-request audit log). Most systems conflate these, which creates problems when users unfriend and then one person re-sends a request.

```typescript
// Canonical relationship. user_a_id < user_b_id enforced at application layer.
// Unique constraint on (user_a_id, user_b_id). Persists via soft-delete.
interface Friendship {
  id: string
  user_a_id: string
  user_b_id: string
  status: "active" | "removed"
  created_at: Date
  updated_at: Date
}

// One row per request. Directional. Multiple requests between the same pair are allowed.
interface FriendRequest {
  id: string
  from_id: string
  to_id: string
  status: "pending" | "accepted" | "declined" | "withdrawn"
  created_at: Date
  updated_at: Date
}
```

### Why two tables?

The `Friendship` row is the source of truth for whether two users are currently friends. The `FriendRequest` rows form a complete, ordered log of every request ever sent between that pair.

When two users unfriend and one sends a new request, the system:
1. Appends a new `FriendRequest` row (no duplicate check on old rows, only on currently `pending` ones)
2. On acceptance, upserts the `Friendship` row via `ON CONFLICT DO UPDATE` — reactivating the existing row rather than creating a duplicate

This means the pair always has exactly one `Friendship` row between them, with a full request history alongside it.

<Note>
  The `user_a_id < user_b_id` ordering is enforced at the application layer, not the database. This canonical ordering ensures the unique constraint on `(user_a_id, user_b_id)` works regardless of which user initiates actions.
</Note>

## Friend request states

`FriendRequest.status` tracks the state of each individual request:

| Status | Description |
|---|---|
| `pending` | Sent, awaiting a response from the recipient |
| `accepted` | Recipient accepted; friendship was created or reactivated |
| `declined` | Recipient declined the request |
| `cancelled` | Sender cancelled before the recipient responded |
| `withdrawn` | Sender explicitly withdrew the request |

## Friendship states

`Friendship.status` describes the current state of the canonical relationship row:

| Status | Description |
|---|---|
| `active` | Users are currently friends |
| `removed` | Users have unfriended; row persists for re-add |

The friendship row is never deleted — only its status changes. This is the soft-delete pattern that makes the re-add flow work cleanly.

## Withdrawal behaviour

A sender can withdraw a pending request before the recipient responds. Withdrawn requests disappear from the recipient's view immediately — `GET /requests/received` filters to `status = 'pending'` only, so withdrawn requests are never shown to the recipient.

The sender can re-send a request after withdrawing because the duplicate check only blocks an actively `pending` request between the same pair. A `withdrawn` request in the log does not count as a blocker.

## Privacy masking rule

When a sender views their sent requests, `declined` statuses are masked and returned as `pending`. This prevents the sender from inferring that they were rejected, which protects the recipient's ability to decline without social awkwardness.

`withdrawn` statuses are shown as-is in the sender's view — since the sender initiated the withdrawal, there is no ambiguity to protect.

<Warning>
  Do not surface raw `declined` statuses to the requesting user. The masking is applied at the API layer, but any client-side display logic should treat the masked `pending` response as final.
</Warning>

## Accept flow

Accepting a friend request upserts the `Friendship` row:

```sql
INSERT INTO friendships (user_a_id, user_b_id, status)
VALUES ($a, $b, 'active')
ON CONFLICT (user_a_id, user_b_id) DO UPDATE SET status = 'active'
```

This single operation handles both the first-time case (inserts a new row) and the re-add case (reactivates an existing `removed` row). The `FriendRequest` row is then marked `accepted`.

## Relationship state in user search

When a user searches for other users, each result includes a `relationship` field that describes the current state between the searcher and the result:

| Value | Meaning |
|---|---|
| `friend` | Users are currently friends |
| `pending_sent` | Current user has sent a request that is pending |
| `pending_received` | Current user has received a request that is pending |
| `none` | No current relationship |

This allows the UI to render the appropriate action (add friend, cancel request, accept/decline, or view profile) without a separate relationship lookup.
