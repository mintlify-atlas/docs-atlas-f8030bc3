A Ktag represents a physical NFC tag issued by Klariti. Tags can exist in inventory before being assigned to a user, so ownership is nullable. The server generates all cryptographic material at registration time — the API client submits only the raw NFC hardware UID, and the server derives every other identifier and signature field.

## Ktag

<ResponseField name="tag_id" type="string" required>
  Server-generated Klariti tag ID. Always prefixed with `kt_`. This is the stable identifier used in API paths and in the NFC payload URL.
</ResponseField>

<ResponseField name="uid_hash" type="string | null">
  A hash of the physical NFC chip's hardware UID (the raw UID/IDm as read by the device). The raw UID is never exposed in API responses — only this derived hash is stored and returned. A unique constraint is enforced on this column, so the same physical chip cannot be registered twice.
</ResponseField>

<ResponseField name="payload" type="string" required>
  The full URL written to the NFC tag: `https://klariti.so/tag/<message>`. The `<message>` segment encodes the signed tag identity in the form `v<sig_version>.<tag_id>.<signature>`. This value is set at registration and is never client-patchable.
</ResponseField>

<ResponseField name="signature" type="string | null">
  Ed25519 signature produced by the server over the string `v<sig_version>|<tag_id>|<uid_hash>`. Used by native iOS clients to verify signed payloads locally against the bundled public key without a network round-trip.
</ResponseField>

<ResponseField name="sig_version" type="number | null">
  Integer version of the signing key and signature scheme used when `signature` was generated. Allows key rotation without invalidating existing tags.
</ResponseField>

<ResponseField name="status" type='"active" | "revoked"' required>
  Current operational state of the tag. Revoking a tag sets `revoked_at` automatically; reactivating it clears `revoked_at`.
</ResponseField>

<ResponseField name="owner_id" type="string | null">
  ID of the user the tag is assigned to. `null` when the tag is in unassigned inventory. Uses `ON DELETE SET NULL` — if the owning user's account is removed, the tag returns to inventory rather than being deleted.
</ResponseField>

<ResponseField name="label" type="string | null">
  A human-readable two-word label generated by the server at registration time (e.g. `"swift-meadow"`). Can be updated by admins via `PATCH /api/admin/ktags/:tag_id`.
</ResponseField>

<ResponseField name="tag_type" type='"WALL" | "MOBILE" | "DESK" | null'>
  Physical form factor of the tag.

  <Expandable title="Tag type values">
    <ResponseField name="WALL" type="string">
      A wall-mounted sticker or plate format tag, typically fixed in a workspace.
    </ResponseField>
    <ResponseField name="MOBILE" type="string">
      A portable tag suitable for a keychain, card, or similar carry format.
    </ResponseField>
    <ResponseField name="DESK" type="string">
      A desk-format tag such as a stand or pad placed at a workstation.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="created_at" type="string | null">
  ISO 8601 datetime. Server-stamped when the tag record is created.
</ResponseField>

<ResponseField name="revoked_at" type="string | null">
  ISO 8601 datetime. Server-stamped automatically when `status` is set to `"revoked"`. Cleared automatically when a revoked tag is reactivated.
</ResponseField>

```ts
interface Ktag {
  tag_id: string              // server-generated, kt_ prefix
  uid_hash: string | null     // hash of the physical NFC chip UID/IDm
  payload: string             // full URL: https://klariti.so/tag/<message>
  signature: string | null    // Ed25519 signature over v<sig_version>|<tag_id>|<uid_hash>
  sig_version: number | null  // signature / key version
  status: "active" | "revoked"
  owner_id: string | null     // nullable — tag may be unassigned inventory
  label: string | null        // server-generated two-word label
  tag_type: "WALL" | "MOBILE" | "DESK" | null
  created_at: string | null
  revoked_at: string | null   // server-stamped on revocation; cleared on reactivation
}
```

## Design notes

### Mutable vs immutable fields

Fields are split into two categories based on whether they can be changed after a tag is registered.

**Admin-patchable** via `PATCH /api/admin/ktags/:tag_id`:

| Field | Notes |
|-------|-------|
| `status` | Revoking stamps `revoked_at`; reactivating clears it |
| `label` | Can be updated or cleared by admins |
| `tag_type` | Can be corrected if a tag was miscategorised |
| `owner_id` | Set to a user ID to assign; `null` to return to inventory |

**Server-managed and never client-patchable:**

| Field | Notes |
|-------|-------|
| `uid_hash` | Derived from the raw UID at registration |
| `payload` | Written to the physical chip; changing it would invalidate tapped tags |
| `signature` | Signed at registration; must match the physical tag |
| `sig_version` | Tied to the signing key used when `signature` was produced |

<Warning>
  `uid_hash`, `payload`, `signature`, and `sig_version` are set once at registration by the server and are never modifiable through any API endpoint. Attempts to patch these fields are rejected.
</Warning>

### Ownership and inventory

`owner_id` references the `user` table with `ON DELETE SET NULL`. If the owning user's account is permanently deleted, the tag row is preserved and `owner_id` is automatically set to `null`, returning the tag to unassigned inventory. This ensures physical tags are not orphaned and can be reassigned.

### NFC payload format

The NFC URL payload encodes the signed tag identity in a single URL-safe segment:

```
https://klariti.so/tag/v<sig_version>.<tag_id>.<signature>
```

The server signs the canonical string `v<sig_version>|<tag_id>|<uid_hash>` with its Ed25519 private key at registration. The resulting signature, together with `sig_version` and `tag_id`, is embedded in the payload URL and also written to the physical chip.

### Tag resolution and verification

Two complementary verification paths exist depending on the client:

**Web** — The landing page resolves a tapped tag by calling `GET /api/tag/:message`. The API parses the signed `<message>`, looks up the matching tag record, and returns public-facing details including the tag name, the owner's name (if assigned), and the current `status`.

**iOS** — The native app reads the raw NFC hardware identifier via Core NFC, then verifies the signed payload locally against the Ed25519 public key bundled at build time. This allows the app to confirm that the physical chip matches the issued record without a network round-trip, and to enforce that a focus session can only be ended by tapping the same physical tag that started it.

<Tip>
  The iOS provisioning flow is a deliberate two-NFC-session process: the app reads first to register or recover the tag record, then starts a second write session to burn the Klariti URL onto the chip if it is blank or mismatched.
</Tip>
